# 2026-07-21

## PGA Best Bets — odds-failure fallback + Grok status correction

Branch `feat/pga-best-bets-page-completion`, cut from `origin/main` at `9e7fb72`.

### Root cause of the "unavailable" artifact
The committed `best-bets.json` was NOT missing or Grok-broken. It was written by
`finalizeSuccessfulArtifact` (proof: it carries `sectionStatus` and
`sourceStatus.article`, which `buildUnavailableArtifact` never emits), with a full
preview and 8-section article but all four pick arrays empty.

Chain:
1. `ODDS_API_KEY` was set in CI — forced by the evidence, since the unset path
   (`generate-pga-best-bets.mjs` old lines 900-902) publishes picks unfiltered.
2. `fetchOdds` returned `{}`. It has four silent-`{}` exits: no sport key, non-array
   response, no bookmakers, or a thrown error.
3. `main` saw the key present and ran `filterByValueAndOdds` on all four markets.
4. That filter drops any pick without a price (`if (!odds) return null`), so an empty
   lookup emptied every market.
5. The article was then generated from the POST-filter arrays, which is why it
   truthfully said "No outright targets were generated this week." The article was
   never wrong — it was describing an already-emptied pick set.
6. Zero sections → `status: "unavailable"`, `reason: "NO_VALID_PICKS"`.

The defect was an asymmetry: odds were optional enrichment with no key, but a hard
precondition with a broken key. The degraded path was strictly worse than the no-key path.

### Lesson — misleading diagnostics cost real time
`sourceStatus.grok: "invalid-response"` was derived purely from post-filter pick count
and never observed an actual Grok failure. Grok worked perfectly. That single mislabel
is why this arrived framed as a Grok/artifact problem. **Never derive a provider's health
from a downstream count that another provider can zero out.**

### Changes
- `scripts/generate-pga-best-bets.mjs` — added exported `selectPublishedPicks`
  (whole-lookup fallback only, no per-market); warning when a key is configured but the
  lookup is empty; `grokStatus` measured pre-odds-filter and emitted on the payload.
- `scripts/run-pga-best-bets-safe.mjs` — `finalizeSuccessfulArtifact` now prefers the
  generator's recorded `sourceStatus.grok`, keeping count-based inference only as a
  legacy fallback.
- Tests added to `generatePgaBestBets.test.ts` (5) and `pgaBestBetsSafeRunner.test.ts` (2).

Deliberately NOT done: per-market fallback, frontend `isEmpty`/article-only rendering,
any change to value thresholds, model ranking, or pick selection.

### Tooling gotchas
- The Grep tool rendered `100 / (n + 100)` as `100 \ (n + 100)`, which looked like a
  syntax error. `node --check` proved the file clean. **Verify apparent syntax bugs
  against raw bytes before "fixing" working code.**
- `public/data/pga/pga-odds.json` is untracked and not gitignored — written only in the
  ephemeral CI runner. Its local absence proves nothing about a CI run.
- The shared `travelerModel` test fixture has no `courseName`; anything reaching
  `buildSummary` → `findWeightEntry` needs one or it throws on `undefined.normalize`.

### Still required
A post-merge regeneration is still needed. This PR fixes the pipeline; it does not
rewrite the currently committed unavailable artifact.

Regeneration does NOT depend on the odds provider recovering. If the lookup is still
empty, the fixed pipeline publishes validated model picks with `odds: null` and
`sourceStatus.odds: "unavailable"`. Provider recovery only restores prices and
value-ranking — never whether picks publish at all.

Verified by finalizing a real dry-run payload: `status: available`, `reason: null`,
`sourceStatus: {model: available, grok: available, odds: unavailable}`, with 4 markets
published and 0 priced. Retaining unpriced picks never falsely reports odds as available.
